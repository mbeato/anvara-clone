## Architecture Patterns

### Pattern 1: Server Component + Client Leaf

The correct pattern already in use throughout this codebase:

```typescript
// page.tsx ā€” Server Component
export default async function DealsPage() {
  const deals = getMockDeals()   // sync or async data
  return <DealsPipeline deals={deals} />  // pass to client leaf
}

// deals-pipeline.tsx ā€” WRONG (currently)
"use client"  // not needed ā€” no hooks, no state, no browser APIs
export function DealsPipeline({ deals }: DealsPipelineProps) {
  return <div>...</div>  // pure rendering
}

// deals-pipeline.tsx ā€” CORRECT
// (no "use client" directive)
export function DealsPipeline({ deals }: DealsPipelineProps) {
  return <div>...</div>  // Server Component ā€” zero client JS added
}
```

### Pattern 2: Legitimate useEffect (UI Effects, Not Data)

These are correct and should NOT be changed:

```typescript
// Scroll-to-bottom ā€” correct use of useEffect for UI side effects
useEffect(() => {
  if (scrollRef.current) {
    scrollRef.current.scrollTop = scrollRef.current.scrollHeight
  }
}, [messages, isLoading])

// Window event listener ā€” correct use of useEffect for browser API
useEffect(() => {
  const onScroll = () => setScrolled(window.scrollY > window.innerHeight * 0.85)
  window.addEventListener("scroll", onScroll, { passive: true })
  return () => window.removeEventListener("scroll", onScroll)
}, [defaultLight])
```

### Pattern 3: Why Removing `"use client"` from Display Components Matters

When a Server Component renders a child with `"use client"`, that child's entire module subtree is included in the client bundle. Removing `"use client"` from `deals-pipeline.tsx` and `campaigns-table.tsx`:
- Moves them to the server ā€” they render HTML on the server, zero client JS
- Reduces client bundle size (small but correct signal)
- Correctly signals to Next.js that these components have no interactivity

### Anti-Patterns to Avoid

- **Anti-pattern:** Adding `"use client"` to any component that imports a client component (unnecessary ā€” only needed if the component itself uses hooks/browser APIs)
- **Anti-pattern:** Removing `"use client"` from components that use `useState`, `useEffect`, `useRouter`, `useRef` for DOM manipulation ā€” these genuinely need it
- **Anti-pattern:** Treating `useRef` alone as a reason to add `"use client"` ā€” `useRef` used only for imperative scroll is borderline; the button `onClick` in `ListingStrip` is the real reason it needs `"use client"`

## Common Pitfalls

### Pitfall 1: Removing `"use client"` From Components That Import Client Components

**What goes wrong:** Removing `"use client"` from a component that imports a component with `"use client"` does NOT cause errors ā€” Next.js handles this correctly. But removing it from a component that itself uses hooks DOES cause errors.

**Why it happens:** Developers conflate "imports a client component" with "needs to be a client component." Only components that directly call hooks or browser APIs need `"use client"`.

**How to avoid:** Check the component's own code, not its imports. If the component file has no `useState`, `useEffect`, `useRef` (for DOM), `useRouter`, `usePathname`, or browser API calls ā€” it doesn't need `"use client"`.

**Warning signs:** TypeScript error "Hooks can only be called inside a function component" ā€” means you removed `"use client"` from something that actually needed it.

### Pitfall 2: The Filter Bar `useEffect` Looks Like Data Fetching

**What goes wrong:** `filter-bar.tsx` has `useEffect(() => { setLiveRange([...]) }, [currentFilters.minPrice, currentFilters.maxPrice])`. This could look like a bug at first glance.

**Why it's actually correct:** This syncs derived local slider display state with URL-driven props when they change externally (e.g., user clicks "clear filters" from `ActiveFilterChips`). Without it, the slider would not reset visually when filters are cleared from elsewhere. This is a controlled component sync pattern, not data fetching.

**How to avoid confusion:** The phase success criteria says "No page-level `useEffect` for data fetching" ā€” this hook is in a leaf display component, not a page, and it doesn't fetch data. It's not a violation.

### Pitfall 3: Overzealous Cleanup Breaking Animation Components

**What goes wrong:** The landing page has multiple client components that use `useEffect` for animations (scroll-triggered counters, typewriter effects, rAF loops, intersection observers). These could be mistakenly flagged as "unnecessary."

**Why it happens:** The success criteria's first bullet ā€” "No page-level `useEffect` for data fetching" ā€” is specifically about data fetching on pages. Animation `useEffect` on leaf components is correct React.

**How to avoid:** Only audit `useEffect` for data fetching (fetch calls, await, loading state). Animation/DOM/event listener effects are correct.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|-----------------|--------------|--------|
| `useEffect` + fetch for page data | `async` Server Component with `await` | Next.js 13+ App Router | Server renders data, zero waterfall |
