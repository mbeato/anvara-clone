## Summary

This phase is an internal quality refactoring pass. The audit below is based on direct inspection of every `.tsx` and `.ts` file outside `node_modules`, `.next`, and `components/ui/` (ShadCN-generated files that should not be touched). The codebase currently builds cleanly with no TypeScript errors (`npm run build` passes, `tsc --noEmit` passes). The phase must not break this.

The primary violations found are: (1) several landing-page components carry `"use client"` solely because they import from other client components or use animation APIs, but a handful could be converted to Server Components; (2) `useEffect` is used extensively in landing-page components for scroll-driven animations and intersection observers — these are **legitimate client-side concerns** and must not be eliminated, just audited; (3) a small set of components have `"use client"` unnecessarily (`for-brands-rightsholders.tsx`, `faq-section.tsx`); (4) no Server Actions use `useFormState` (deprecated) — none use `useActionState` either; (5) no `any` types exist in production code (only a comment); (6) `import type` is used inconsistently but TypeScript strict mode is already on.

**Primary recommendation:** This phase is mostly about validating existing patterns are correct and fixing the small number of actual violations. It is NOT a large refactor — the codebase is already mostly well-structured.

**Key finding on `browse-client.tsx` `useEffect`:**

```tsx
// BROWSE-09: 200-400ms skeleton loading delay
const [isLoading, setIsLoading] = useState(true);
useEffect(() => {
  const timer = setTimeout(() => setIsLoading(false), 300);
```

This is an artificial skeleton delay. The route already has a `loading.tsx` with skeletons that fires during SSR/navigation. ### 4. `import type` Enforcement Audit

The project does NOT have `verbatimModuleSyntax: true` or `importsNotUsedAsValues: "error"` in `tsconfig.json`, so the TypeScript compiler does not enforce `import type`. However, the coding standard requires it for type-only imports.

**Current state — `import type` IS used correctly in:**
- `app/(app)/browse/_components/browse-client.tsx` — `import type { getAllProperties }`
- `app/(app)/browse/_components/recommendations-strip.tsx` — `import type { getAllProperties }`
- `app/(app)/browse/_components/property-card.tsx` — `import type { getAllProperties }`
- `components/messaging/thread-list.tsx`, `thread-list-item.tsx`, `messaging-client.tsx`, `conversation-view.tsx` — `import type { getThreads }` / `import type { getThread }`
- `components/theme-provider.tsx` — `import type { ComponentProps }`
- `app/layout.tsx` — `import type { Metadata }`
- `components/property-detail/property-demographics.tsx` — `import type { ChartConfig }`

**Where `import type` enforcement could be added:**

The existing usage is good. The codebase already identifies type-only imports from data layer functions. The standard should be enforced via `verbatimModuleSyntax: true` in tsconfig to make violations compile errors. However, enabling this requires confirming it doesn't break existing code.

**Action:** Add `"verbatimModuleSyntax": true` to `tsconfig.json` after confirming no existing imports break.

### 5. Server Actions Audit (`useActionState` vs `useFormState`)

**Findings:**
- `app/actions/message.ts` — `sendMessage()` — called directly with `await` from client component, NOT via form action ✓
- `app/actions/offer.ts` — `createOffer()`, `createOfferInThread()` — called directly with `await` from client components ✓
- Neither `useFormState` nor `useActionState` is used anywhere in the codebase

**Verdict:** The server actions are called imperatively (not bound to forms via `action=` prop), which means `useActionState` does not apply to the current patterns. No migration needed. If forms are refactored to use native form submission, `useActionState` (React 19 API) should be used — NOT `useFormState` (deprecated react-dom import).

**React 19 pattern for future reference:**
```tsx
// CORRECT (React 19)
import { useActionState } from "react";
const [state, formAction, isPending] = useActionState(myServerAction, initialState);

// WRONG (deprecated)
import { useFormState } from "react-dom"; // no longer exists in React 19
```

## Architecture Patterns

### React 19 Server Component Pattern (CORRECT in this codebase)

```tsx
// app/(app)/browse/page.tsx — CORRECT pattern
export default async function BrowsePage({ searchParams }: { searchParams: SearchParams }) {
  const params = await searchParams;   // Next.js 15: searchParams is a Promise
  const properties = await getAllProperties();
  return <BrowseClient properties={properties} />;
}
```

### `'use client'` Boundary Rule

```
page.tsx           — Server Component (never has 'use client')
layout.tsx         — Server Component (never has 'use client')
_components/       — Server by default; 'use client' only if interactive
components/        — Mixed; 'use client' only on leaf interactive components
```

### Correct `import type` Pattern

```tsx
// CORRECT — type-only import uses 'import type'
import type { Metadata } from "next";
import type { getThreads } from "@/lib/data";

// CORRECT — value import (components, functions) uses regular import
import { Button } from "@/components/ui/button";
import { getAllProperties } from "@/lib/data";
```

### `useActionState` Pattern for Future Forms (React 19)

```tsx
"use client"
import { useActionState } from "react";  // React 19 — from "react", not "react-dom"

const [state, formAction, isPending] = useActionState(
  async (prevState: FormState, formData: FormData) => {
    // action logic
    return { success: true };
  },
  { success: false }
);
```

## Common Pitfalls

### Pitfall 1: Removing `"use client"` from a component that wraps client components

**What goes wrong:** Removing `"use client"` from a parent thinking it's safe, then getting a runtime error because the parent receives a client component as a prop or children.

**Prevention:** Before removing `"use client"`, verify that ALL children and all imports are either Server-safe or themselves carry `"use client"`. The parent does NOT need `"use client"` if it only renders children that carry it — but it CANNOT use hooks, event handlers, or browser APIs.

**This applies to:** `for-brands-rightsholders.tsx` and `faq-section.tsx` — safe to remove because they use no hooks and their children carry their own directives.

### Pitfall 2: Confusing `useFormState` removal with `useActionState`

**What goes wrong:** Migrating from `useFormState` (deprecated `react-dom` export) to `useActionState` (React 19 `react` export) but forgetting to change the import path.

**Prevention:** `useActionState` is imported from `"react"` in React 19, NOT `"react-dom"`. The current codebase uses neither, so this is only relevant if future forms are added.

### Pitfall 3: `verbatimModuleSyntax` breaking existing code

**What goes wrong:** Enabling `verbatimModuleSyntax: true` when re-exporting types without `export type` syntax causes compiler errors.

**Prevention:** After adding `verbatimModuleSyntax: true`, run `tsc --noEmit` and fix any reported errors before committing. Common fix: change `import { SomeType }` to `import type { SomeType }` where the value is only used as a type.

### Pitfall 4: Removing `isLoading` skeleton from `browse-client.tsx` causing layout shift

**What goes wrong:** The `loading.tsx` fires during navigation (Suspense boundary), but after mount, removing the client-side skeleton means the grid renders immediately — which is correct. However, if `loading.tsx` was not wired up, removing the client skeleton would cause a flash of unstyled content.

**Prevention:** Verify `loading.tsx` exists (it does — at `app/(app)/browse/loading.tsx`) before removing the client-side `useEffect`/`isLoading` logic.
