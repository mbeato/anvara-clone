## Summary

Phase 6.1 adds an analytics dashboard route at `/dashboard` — a page already referenced in the sidebar nav but not yet implemented. The stack for this phase is entirely pre-installed: Recharts 2.15.4 is in `package.json`, the ShadCN `chart.tsx` wrapper (`ChartContainer`, `ChartTooltip`, `ChartTooltipContent`) is already in `components/ui/chart.tsx`, the ShadCN `Card` family is in `components/ui/card.tsx`, and the sidebar nav already has a "Dashboard" entry pointing to `/dashboard`. No new dependencies are required.

The standard approach is: (1) a Server Component `page.tsx` at `app/(app)/dashboard/page.tsx` that queries Prisma for real seed data, (2) pure derivation of analytics-like metrics from that data in a `lib/data/analytics.ts` helper, (3) chart components in `app/(app)/dashboard/_components/` that are `'use client'` leaves (required by Recharts), and (4) KPI cards and a recent activity table that are pure Server Component markup. All data generation stays deterministic and seeded — no random values at render time.

The existing seed has 100 properties across 5 categories (sports: 27, lifestyle: 23, music: 18, food: 18, arts: 14), 5 threads with messages and 2 offers, and rich property data (priceFrom, audienceTotalReach, category). This is enough to derive: total portfolio reach, total deal value, offer counts, message counts, category breakdowns for bar charts, and time-series data generated from `createdAt` timestamps spread with deterministic offsets.

**Primary recommendation:** Use `ChartContainer` + Recharts `AreaChart` for impressions trend, `BarChart` for category spend breakdown, ShadCN `Card` for KPI widgets, and a plain `<table>` for recent activity — all as a single Server Component page with isolated `'use client'` chart leaves.

### Pattern 1: Server Component Page with 'use client' Chart Leaves

**What:** `page.tsx` is `async` (no `'use client'`), calls `getAllProperties()` and `getThreads()` via `Promise.all`, derives all metrics in helpers, then passes serializable data (plain arrays/objects) as props to child components. Chart components are `'use client'` only because Recharts requires it — they receive pre-computed data arrays as props.

**When to use:** Always for this dashboard. Recharts components require client-side rendering (they use `useEffect` and `ResizeObserver` internally), but the data derivation and KPI math does not need to be client-side.

**Example (page.tsx pattern):**
```typescript
// app/(app)/dashboard/page.tsx
// NO 'use client' — this is a Server Component
import { getAllProperties, getThreads } from "@/lib/data"
import { deriveAnalytics } from "@/lib/data/analytics"
import { ImpressionsAreaChart } from "./_components/impressions-area-chart"
import { CategoryBarChart } from "./_components/category-bar-chart"
import { KpiCard } from "./_components/kpi-card"
import { RecentActivityTable } from "./_components/recent-activity-table"

export default async function DashboardPage() {
  const [properties, threads] = await Promise.all([
    getAllProperties(),
    getThreads(),
  ])

  const analytics = deriveAnalytics(properties, threads)

  return (
    <div className="space-y-6 max-w-7xl mx-auto">
      {/* KPI row */}
      <div className="grid grid-cols-2 lg:grid-cols-4 gap-4">
        {analytics.kpis.map((kpi) => (
          <KpiCard key={kpi.label} {...kpi} />
        ))}
      </div>
      {/* Charts row */}
      <div className="grid grid-cols-1 lg:grid-cols-3 gap-4">
        <div className="lg:col-span-2">
          <ImpressionsAreaChart data={analytics.impressionsTrend} />
        </div>
        <CategoryBarChart data={analytics.categoryBreakdown} />
      </div>
      {/* Recent activity */}
      <RecentActivityTable offers={analytics.recentOffers} threads={threads} />
    </div>
  )
}
```

### Pattern 2: Deterministic Fake Time-Series from Seed Data

**What:** The seed has `property.createdAt` and `thread.createdAt` timestamps, but they cluster near seed time. To create a believable "30-day impressions trend," generate a deterministic array of 30 daily data points using a seeded pseudo-random approach based on `properties.length` and `totalReach` — not `Math.random()`. This produces stable output on every render.

**Why deterministic matters:** If you use `Math.random()` in a Server Component, the values change on every request and React's hydration reconciliation may diverge. Use a simple seeded generator.

**Example (analytics helper):**
```typescript
// lib/data/analytics.ts
// No Prisma imports — takes pre-fetched data as arguments

type Property = { priceFrom: number; audienceTotalReach: number; category: string; name: string }
type Thread = { createdAt: Date; offers: { amount: number; status: string }[]; property: Property }

export function generateImpressionsTrend(
  totalReach: number,
  dayCount = 30
): Array<{ date: string; impressions: number }> {
  // Seeded deterministic variation — no Math.random()
  const base = Math.floor(totalReach / 1000) // scale to ~thousands per day
  return Array.from({ length: dayCount }, (_, i) => {
    const seed = (base * (i + 1) * 7919) % 1000 // prime multiply for variation
    const variance = (seed / 1000) * 0.4 - 0.2   // ±20% variance
    const value = Math.floor(base * (1 + variance) * (1 + i * 0.01)) // slight upward trend
    const date = new Date(Date.now() - (dayCount - i - 1) * 86400000)
    return {
      date: date.toLocaleDateString("en-US", { month: "short", day: "numeric" }),
      impressions: value,
    }
  })
}
```

### Pattern 4: KPI Card with Trend Indicator

**What:** A Server Component wrapping ShadCN `Card` with a metric value, label, trend percentage, and TrendingUp/TrendingDown icon. No client code needed.

```typescript
// app/(app)/dashboard/_components/kpi-card.tsx
// NO 'use client'
import { TrendingUp, TrendingDown } from "lucide-react"
import { Card, CardContent, CardHeader, CardTitle } from "@/components/ui/card"
import type { LucideIcon } from "lucide-react"

interface KpiCardProps {
  label: string
  value: string          // pre-formatted: "$2.4M", "143K", "18.3%"
  trend: number          // percentage change e.g. +12.4 or -3.1
  icon: LucideIcon
}

export function KpiCard({ label, value, trend, icon: Icon }: KpiCardProps) {
  const isUp = trend >= 0
  return (
    <Card className="py-4">
      <CardHeader className="pb-2">
        <div className="flex items-center justify-between">
          <CardTitle className="text-sm font-medium text-muted-foreground">{label}</CardTitle>
          <Icon className="h-4 w-4 text-muted-foreground" />
        </div>
      </CardHeader>
      <CardContent>
        <p className="text-2xl font-bold tabular-nums">{value}</p>
        <p className={`flex items-center gap-1 text-xs mt-1 ${isUp ? "text-emerald-600" : "text-red-500"}`}>
          {isUp ? <TrendingUp className="h-3 w-3" /> : <TrendingDown className="h-3 w-3" />}
          {isUp ? "+" : ""}{trend.toFixed(1)}% vs last month
        </p>
      </CardContent>
    </Card>
  )
}
```

### Anti-Patterns to Avoid

- **Putting `'use client'` on `page.tsx`:** This forces the entire page client-side and prevents async data fetching with `await` at the page level. Keep page.tsx as a Server Component.
- **Using `Math.random()` in analytics helpers:** Creates hydration mismatches and inconsistent values across renders. Use deterministic seeded calculation.
- **Calling Prisma directly inside chart components:** Chart components must be `'use client'` for Recharts; `'use client'` components cannot call Prisma. Fetch in the Server Component page and pass serializable data as props.
- **Hardcoding pixel heights on ChartContainer without override:** `ChartContainer` defaults to `aspect-video`. Override with explicit `className="h-64 w-full"` and the `[&_.recharts-wrapper]:!aspect-auto` trick (seen in demographics component) if needed.
- **Importing Recharts in Server Components:** Any file that imports from `recharts` must be `'use client'`. Recharts uses browser APIs.

## Common Pitfalls

### Pitfall 1: Sidebar Already Has "Dashboard" Entry

**What goes wrong:** Developer creates a new sidebar item for analytics instead of using the existing one, creating a duplicate entry.

**Why it happens:** Not reading the existing sidebar code before planning.

**How to avoid:** The existing `app-sidebar.tsx` already has `{ title: "Dashboard", href: "/dashboard", icon: Gauge }`. The dashboard route must be at `app/(app)/dashboard/page.tsx` to match. Do NOT add a new sidebar entry — the existing one already serves this. The success criteria says "analytics-style icon" — since `Gauge` is already there and semantically appropriate, no sidebar change may be needed. Confirm with the existing nav before modifying.

**Warning signs:** Plan steps that modify `app-sidebar.tsx` without first checking current nav items.

### Pitfall 2: Recharts Components Cannot Be Server Components

**What goes wrong:** Importing `AreaChart` from `recharts` in a file without `'use client'` causes a Next.js error: "You're importing a component that needs X. It only works in a Client Component but none of its parents are marked with 'use client'."

**Why it happens:** Recharts uses `useEffect` and DOM APIs internally.

**How to avoid:** Every file that imports from `recharts` or `components/ui/chart.tsx` must have `"use client"` as its first line.

### Pitfall 3: Card Component Default Padding

**What goes wrong:** `Card` has `py-6` built in. Adding extra padding in `CardContent` (which has `px-6`) creates excessive whitespace for compact KPI cards.

**How to avoid:** Override with `className="py-4"` on `Card` for KPI cards. Study the existing `card.tsx` source — `CardHeader` uses `px-6` and `CardContent` uses `px-6`. For tighter KPI layouts, pass `className` overrides.

### Pitfall 4: Serialization of Prisma Return Types as Chart Props

**What goes wrong:** Passing raw Prisma results (which include `Date` objects, nested objects, etc.) directly as props to `'use client'` chart components. Next.js Server-to-Client prop passing requires serializable values.

**Why it happens:** Prisma returns `Date` objects; `Date` is not directly serializable as a prop across the server/client boundary in Next.js App Router.

**How to avoid:** In `analytics.ts`, convert all `Date` values to `string` (e.g., `.toISOString()` or formatted strings) before returning. The chart components receive `Array<{ date: string; value: number }>` — fully serializable.

### Pitfall 5: KpiCard Requires LucideIcon Type Import

**What goes wrong:** `import { LucideIcon } from "lucide-react"` without `type` keyword fails under `verbatimModuleSyntax` (enforced since Phase 5.1).

**How to avoid:** Use `import type { LucideIcon } from "lucide-react"`.

### Deriving KPI Metrics from Seed Data

```typescript
// lib/data/analytics.ts
import type { getAllProperties, getThreads } from "./index"

type Properties = Awaited<ReturnType<typeof getAllProperties>>
type Threads = Awaited<ReturnType<typeof getThreads>>

export interface KpiData {
  label: string
  value: string
  trend: number
  iconName: "ChartArea" | "TrendingUp" | "Briefcase" | "MessageSquare"
}

export function deriveKpis(properties: Properties, threads: Threads): KpiData[] {
  const totalReach = properties.reduce((sum, p) => sum + p.audienceTotalReach, 0)
  const totalPortfolioValue = properties.reduce((sum, p) => sum + p.priceFrom, 0)
  const allOffers = threads.flatMap((t) => t.offers)
  const totalOfferValue = allOffers.reduce((sum, o) => sum + o.amount, 0)
  const acceptedOffers = allOffers.filter((o) => o.status === "accepted").length
  const conversionRate = allOffers.length > 0
    ? (acceptedOffers / allOffers.length) * 100
    : 12.4 // fallback when no accepted offers in seed

  return [
    {
      label: "Total Impressions",
      value: formatReach(totalReach),
      trend: 14.2,
      iconName: "ChartArea",
    },
    {
      label: "Active Campaigns",
      value: threads.length.toString(),
      trend: 8.1,
      iconName: "Briefcase",
    },
    {
      label: "Conversion Rate",
      value: `${conversionRate.toFixed(1)}%`,
      trend: 2.3,
      iconName: "TrendingUp",
    },
    {
      label: "Deal Pipeline",
      value: `$${(totalOfferValue / 1000).toFixed(0)}K`,
      trend: 22.7,
      iconName: "MessageSquare",
    },
  ]
}

function formatReach(n: number): string {
  if (n >= 1_000_000) return `${(n / 1_000_000).toFixed(1)}M`
  if (n >= 1_000) return `${(n / 1_000).toFixed(0)}K`
  return n.toString()
}
```

### Category Breakdown for BarChart

```typescript
export interface CategoryStat {
  category: string
  properties: number
  totalReach: number
  avgPrice: number
}

export function deriveCategoryBreakdown(properties: Properties): CategoryStat[] {
  const byCategory = new Map<string, Properties>()
  for (const p of properties) {
    const arr = byCategory.get(p.category) ?? []
    arr.push(p)
    byCategory.set(p.category, arr)
  }
  return Array.from(byCategory.entries()).map(([category, props]) => ({
    category: category.charAt(0).toUpperCase() + category.slice(1),
    properties: props.length,
    totalReach: props.reduce((sum, p) => sum + p.audienceTotalReach, 0),
    avgPrice: Math.floor(props.reduce((sum, p) => sum + p.priceFrom, 0) / props.length),
  }))
}
// Result: 5 categories — Sports (27), Lifestyle (23), Music (18), Food (18), Arts (14)
```
