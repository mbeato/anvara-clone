# Phase 6.5: Accessibility and Animation Hardening - Research

**Researched:** 2026-02-23
**Domain:** Motion/Framer Motion v12 optimization, CSS accessibility, mobile navigation
**Confidence:** HIGH

## Summary

Phase 6.5 is a quality hardening pass on the landing page with four distinct concerns: (1) bundle size reduction via LazyMotion, (2) prefers-reduced-motion accessibility, (3) mobile hamburger menu, and (4) animation duration capping at 500ms.

All libraries are already installed — Motion v12.34.3, Lucide React, ShadCN Sheet. No new packages are needed. The work is surgical: two landing components use `motion/react` directly (`section-reveal.tsx` and `hero-section.tsx`), and five components have animation durations exceeding 500ms. The navbar already has nav links hidden below `lg` breakpoint with no hamburger substitute — that is the mobile gap to fix.

The standard approach is: wrap the landing page in a thin client `LandingProviders` component that provides both `LazyMotion` (for bundle optimization) and `MotionConfig reducedMotion="user"` (for Motion-based accessibility), then separately add a CSS `@media (prefers-reduced-motion: reduce)` block in `globals.css` to disable CSS keyframe animations (marquee, pulse) for users with reduced motion preference. CSS inline-style transitions in `anvara-intelligence.tsx` and `performance-reporting.tsx` need their own `useReducedMotion` hook to be conditionally disabled.

**Primary recommendation:** Create a `LandingProviders` client wrapper with `LazyMotion + MotionConfig`, migrate `motion.div` to `m.div` in the two landing components, add CSS media query for keyframe animations, fix durations in 5 files, and add hamburger Sheet to the navbar.

### Pattern 1: LazyMotion + m Component Migration

**What:** Replace `motion.div` with `m.div` and wrap with `LazyMotion features={domAnimation}`
**When to use:** Any time Motion components exist on a page

Import path for `m` is `motion/react-m` (a separate subpath package), NOT from `motion/react`:

```typescript
// Source: verified from node_modules/motion/package.json exports map
// LandingProviders.tsx (new "use client" component)
"use client"

import { LazyMotion, domAnimation, MotionConfig } from "motion/react"

export function LandingProviders({ children }: { children: React.ReactNode }) {
  return (
    <LazyMotion features={domAnimation} strict>
      <MotionConfig reducedMotion="user">
        {children}
      </MotionConfig>
    </LazyMotion>
  )
}
```

```typescript
// section-reveal.tsx — AFTER migration
"use client"
import * as m from "motion/react-m"

export function SectionReveal({ children, delay = 0, className }: SectionRevealProps) {
  return (
    <m.div
      initial={{ opacity: 0, y: 32 }}
      whileInView={{ opacity: 1, y: 0 }}
      viewport={{ once: true, amount: 0.15 }}
      transition={{ duration: 0.4, ease: [0.25, 0.1, 0.25, 1], delay }}
      className={className}
    >
      {children}
    </m.div>
  )
}
```

```typescript
// hero-section.tsx — AFTER migration (AnimatePresence stays from motion/react)
"use client"
import { AnimatePresence } from "motion/react"
import * as m from "motion/react-m"

// Replace motion.div with m.div — all props are identical
```

**Critical note on `strict` prop:** Using `strict={true}` on `LazyMotion` will throw a runtime error if any `motion.div` is accidentally left in the tree. Use it during development to catch missed migrations.

**Critical note on `import * as m`:** With `verbatimModuleSyntax` enabled in tsconfig, named imports work fine. The `import * as m` pattern gives access to `m.div`, `m.span`, `m.section`, etc.

### Pattern 2: MotionConfig reducedMotion="user"

**What:** Single wrapper that tells Motion to respect the OS prefers-reduced-motion setting
**Behavior:** When `prefers-reduced-motion: reduce` is active, Motion disables transform/layout/viewport animations. Opacity and color animations still play.

```typescript
<MotionConfig reducedMotion="user">
  {/* All motion.div / m.div descendants respect system preference */}
</MotionConfig>
```

**Scope:** Only affects Motion components (`m.div`, `motion.div`). Does NOT affect:
- CSS `transition-*` utility classes
- CSS `animation-*` / `@keyframes` animations
- Inline `style={{ transition: "..." }}` elements

### Pattern 3: CSS prefers-reduced-motion for Keyframe Animations

**What:** CSS media query to disable CSS-only animations
**When to use:** For `animate-pulse`, `animate-[marquee_30s...]`, and the `@keyframes marquee` CSS animations that MotionConfig cannot reach.

```css
/* Add to globals.css */
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
```

**Warning:** This broad rule will also zero-out Motion-controlled transitions since Motion uses inline styles under the hood. Test with MotionConfig first — the broad CSS rule is a fallback safety net for CSS-only animations.

**Alternative (targeted):** Instead of the broad rule, target specific CSS class names:
```css
@media (prefers-reduced-motion: reduce) {
  [class*="animate-"] {
    animation: none !important;
  }
}
```

### Pattern 4: useReducedMotion for JS-Controlled CSS Transitions

**What:** Hook that reads `window.matchMedia('(prefers-reduced-motion: reduce)')`
**When to use:** In `anvara-intelligence.tsx` and `performance-reporting.tsx` where transitions are applied via `style={{ transition: "...", transitionDelay: "..." }}` on regular `<div>` elements — MotionConfig cannot reach these.

```typescript
// Source: verified from node_modules/motion/react exports
"use client"
import { useReducedMotion } from "motion/react"

function AnimatedFlow() {
  const reducedMotion = useReducedMotion()

  // When reducedMotion is true, skip the intersection observer animation
  // and show all elements at full opacity immediately
  const visible = reducedMotion ? true : localVisibleState

  return (
    <div
      style={{
        opacity: visible ? 1 : 0,
        // When reducedMotion, don't apply transition at all
        transition: reducedMotion ? undefined : "opacity 0.4s ease",
      }}
    />
  )
}
```

### Pattern 6: Animation Duration Hardening

**What:** Reduce all durations > 500ms to 300–400ms range
**When to use:** At each animated component

| File | Current Value | Fixed Value | Type |
|------|--------------|-------------|------|
| `section-reveal.tsx` | `duration: 0.55` | `duration: 0.4` | Motion transition |
| `hero-section.tsx` | `transition-opacity duration-1000` | `transition-opacity duration-300` | Tailwind CSS class |
| `what-is-anvara.tsx` | `duration-700` (×3) | `duration-300` | Tailwind CSS class |
| `anvara-intelligence.tsx` | `stroke-dashoffset 0.6s` | `stroke-dashoffset 0.4s` | Inline CSS transition |
| `anvara-intelligence.tsx` | `transitionDuration: "0.6s"` | `transitionDuration: "0.4s"` | Inline CSS transition |
| `performance-reporting.tsx` | `const dur = "1.2s"` | `const dur = "0.5s"` | Inline CSS transition (SVG) |

**Note on delays:** Delays (e.g., `transitionDelay: "1.3s"`) are NOT durations and do NOT violate the requirement. They control when an animation starts, not how long it runs. These can remain unchanged.

**Note on Sheet:** `SheetContent` has `data-[state=open]:duration-500` which is exactly 500ms — this does not exceed 500ms so it is acceptable per the success criteria ("exceeds 500ms").

### Anti-Patterns to Avoid

- **Leaving `motion.div` in a `LazyMotion strict` tree:** Will throw runtime error. Migration must be complete before enabling `strict`.
- **Using `motion/react` for the `m` component:** The `m` component comes from `motion/react-m` (separate subpath). Importing it from `motion/react` would import the full bundle, defeating the purpose.
- **Applying `reducedMotion="always"` instead of `"user"`:** `"always"` disables motion for all users regardless of OS setting. Use `"user"` to respect the preference.
- **Applying the broad CSS `*` reduced-motion rule without understanding that it also affects Motion:** Motion renders CSS transitions via inline styles in some cases. Test after adding the CSS rule.
- **Adding LazyMotion to root layout:** Since Motion is only used on the landing page, LazyMotion belongs in a landing-specific wrapper, not `app/layout.tsx`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Mobile nav focus trap | Custom focus trap logic | ShadCN Sheet (Radix Dialog) | Radix handles focus trap, keyboard navigation, overlay dismiss, ARIA roles automatically |

## Common Pitfalls

### Pitfall 1: AnimatePresence must still come from motion/react
**What goes wrong:** Importing `AnimatePresence` from `motion/react-m` fails because it doesn't exist there.
**Why it happens:** `motion/react-m` only exports motion component proxies (`m.div`, `m.span`, etc.), not orchestration utilities.
**How to avoid:** Keep `import { AnimatePresence } from "motion/react"` alongside `import * as m from "motion/react-m"`.
**Warning signs:** TypeScript error "Module has no exported member 'AnimatePresence'" when importing from `motion/react-m`.

### Pitfall 2: verbatimModuleSyntax and motion imports
**What goes wrong:** TypeScript error when using `import * as m from "motion/react-m"` with `verbatimModuleSyntax` enabled.
**Why it happens:** `verbatimModuleSyntax` requires type-only imports to use `import type`. However, `motion/react-m` exports runtime values (components), not types, so `import * as m` is correct.
**How to avoid:** Only use `import type` for pure type imports. Runtime value imports like `import * as m from "motion/react-m"` are correct as-is.

### Pitfall 3: LazyMotion strict catches leftover motion.div
**What goes wrong:** With `strict={true}`, any component in the tree that still uses `motion.div` (not `m.div`) will throw a runtime error in development.
**Why it happens:** The ShadCN component files in `components/ui/` do not use Motion, but if any other component imports `motion.div` from `motion/react` within the LazyMotion boundary, it will error.
**How to avoid:** Do a complete search for `motion.div`, `motion.span`, etc. before enabling `strict`. Only the landing components (`section-reveal.tsx`, `hero-section.tsx`) use Motion — confirm no others exist.
**Warning signs:** Console error "You have used a motion component within a LazyMotion tree..."

### Pitfall 4: MotionConfig does not cover inline CSS transitions
**What goes wrong:** CSS-transition-based components (`anvara-intelligence.tsx`, `performance-reporting.tsx`, `what-is-anvara.tsx`) still animate even when `prefers-reduced-motion` is on.
**Why it happens:** `MotionConfig reducedMotion="user"` only affects Motion's own animation engine, not browser-rendered CSS transitions or animations.
**How to avoid:** Use `useReducedMotion()` in those components to conditionally remove the `transition` property from inline styles when reduced motion is preferred. Also add CSS `@media (prefers-reduced-motion: reduce)` in globals.css.

### Pitfall 5: LandingProviders must be "use client" but page.tsx is a Server Component
**What goes wrong:** Adding `LazyMotion` and `MotionConfig` directly to `app/page.tsx` (Server Component) causes a build error because these are client-side React components.
**Why it happens:** `LazyMotion` and `MotionConfig` use React context which requires client rendering.
**How to avoid:** Create a separate `landing-providers.tsx` file with `"use client"` at the top, and import it into the Server Component `page.tsx`. Server Components CAN render client components as children.

### Pitfall 6: The marquee animation in hero-section.tsx is CSS, not Motion
**What goes wrong:** The scrolling marquee (`animate-[marquee_30s_linear_infinite]`) is a Tailwind CSS animation, so `MotionConfig` has no effect on it.
**Why it happens:** The marquee is declared as `@keyframes marquee` in `globals.css` and applied via Tailwind's `animate-[...]` utility.
**How to avoid:** Add the `@media (prefers-reduced-motion: reduce)` CSS rule to `globals.css` to disable these CSS animations for users with the preference.

## Open Questions

1. **Should `LazyMotion strict` be enabled?**
   - What we know: `strict=true` throws runtime errors if `motion.*` components exist in the tree
   - What's clear: Only `section-reveal.tsx` and `hero-section.tsx` use Motion components — confirmed by grep
   - Recommendation: Enable `strict={true}` as a safety net. Since only 2 files use Motion and both will be migrated, no risk.

2. **Scope of CSS @media rule in globals.css**
   - What we know: A broad `* { animation-duration: 0.01ms }` rule may interfere with ShadCN/Radix animations
   - What's unclear: Whether the tw-animate-css import has its own reduced-motion handling
   - Recommendation: Use the targeted `[class*="animate-"]` selector instead of the broad `*` selector to avoid unintended side effects on ShadCN components.

3. **What to do with video crossfade in hero-section.tsx**
   - What we know: `transition-opacity duration-1000` on `<video>` elements is a CSS transition, not Motion
   - Requirement: "No animation duration exceeds 500ms" — 1000ms is a clear violation
   - Recommendation: Change to `duration-300`. The video crossfade is decorative and looks fine faster. When `prefers-reduced-motion`, the CSS `@media` rule will zero it out anyway.
