# Phase 6.2: Public Listing Page - Research

**Researched:** 2026-02-22
**Domain:** Next.js App Router route groups, public detail page, gated content UI, landing page integration
**Confidence:** HIGH (all critical claims verified against codebase inspection and official Next.js docs)

---

## Summary

Phase 6.2 builds a public-facing property detail page at `/p/[slug]` that shows a limited view (hero, title, description, category) and gates sensitive sections (demographics, tier pricing, activation formats) behind a "Login to see more" blurred/locked overlay. It reuses the same Prisma seed data and `getPropertyBySlug` query helper as the private detail page — no duplicate data or schema changes needed.

The key architectural challenge is creating a route that opts out of the `(app)` layout group (which wraps everything with the sidebar, header bar, and footer). The solution is a new `(public)` route group at `app/(public)/p/[slug]/page.tsx` with its own minimal `layout.tsx` that includes `LandingNavbar` and `LandingFooter` instead of the sidebar shell. This is exactly how the current project structures `app/page.tsx` (the landing page) vs `app/(app)/` — the root layout is shared, but each route group has its own shell.

The landing page currently has multiple components (`LandingPropertyCard`, `AccessMarket` `PropertyRow`, `FinalCTA` vertical scroll cards) that link to `/listings/${property.id}` (the private route). All of these must be updated to link to `/p/${property.slug}` instead. This is the most cross-cutting change in the phase.

**Primary recommendation:** Create `app/(public)/p/[slug]/` as a new route group with its own `layout.tsx` that renders `LandingNavbar` + `children` + `LandingFooter`. Reuse existing `PropertyHero`, `PropertyMeta`, `PropertyAbout` components unchanged. Build a new `LockedSection` component for the gated content overlay. Update all landing page card links from `/listings/${id}` to `/p/${slug}`.

### Pattern 1: (public) Route Group with Own Layout

**What:** A route group folder named `(public)` that shares no layout with `(app)`. Each group has its own `layout.tsx`. Both inherit from `app/layout.tsx` (the root layout that provides ThemeProvider and fonts).

**When to use:** Any page that must display the landing page chrome (LandingNavbar + LandingFooter) instead of the app sidebar/header.

**Example:**
```typescript
// app/(public)/layout.tsx — NEW FILE
// Source: https://nextjs.org/docs/app/building-your-application/routing/route-groups

export default function PublicLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <div className="force-light min-h-screen bg-background">
      <LandingNavbar />
      <main>{children}</main>
      <LandingFooter />
    </div>
  )
}
```

**Critical:** The `(public)` folder name with parentheses is NOT included in the URL. The route `app/(public)/p/[slug]/page.tsx` resolves to `/p/[slug]`.

**Existing model:** The current project already uses this pattern — `app/page.tsx` is the landing page at `/` and manually includes `LandingNavbar` and `LandingFooter`. The `app/(app)/layout.tsx` provides the sidebar for all `/listings/...`, `/messages/...`, `/dashboard/...` routes. Adding `app/(public)/layout.tsx` follows the same pattern for public routes.

### Pattern 2: Public Detail Page (Server Component)

**What:** Async Server Component that fetches property data and renders visible sections plus locked overlay sections. Same async params pattern as the private detail page.

**When to use:** The public `/p/[slug]` route.

**Example:**
```typescript
// app/(public)/p/[slug]/page.tsx — NEW FILE
import { notFound } from "next/navigation"
import { getPropertyBySlug } from "@/lib/data"
import { PropertyHero } from "@/components/property-detail/property-hero"
import { PropertyMeta } from "@/components/property-detail/property-meta"
import { PropertyAbout } from "@/components/property-detail/property-about"
import { LockedSection } from "@/components/public-listing/locked-section"

export default async function PublicListingPage({
  params,
}: {
  params: Promise<{ slug: string }>
}) {
  const { slug } = await params
  const property = await getPropertyBySlug(slug)

  if (!property) {
    notFound()
  }

  return (
    <div className="max-w-6xl mx-auto px-4 py-6">
      {/* VISIBLE: Hero image */}
      <PropertyMeta property={property} />
      <div className="mt-4">
        <PropertyHero images={[property.imageUrl]} />
      </div>
      <div className="mt-6 space-y-8">
        {/* VISIBLE: About */}
        <PropertyAbout description={property.description} />

        {/* LOCKED: Tier Pricing */}
        <LockedSection title="Sponsorship Tiers" />

        {/* LOCKED: Audience Demographics */}
        <LockedSection title="Audience Demographics" />

        {/* LOCKED: Activation Formats */}
        <LockedSection title="Activation Formats" />
      </div>
    </div>
  )
}

export async function generateStaticParams() {
  return []
}
```

**Note on layout:** The public page does NOT use the sidebar or sticky offer sidebar. The page is single-column (no 5-col grid split). The offer sidebar is part of the authenticated experience.

### Anti-Patterns to Avoid

- **Using property ID in public URLs:** Landing page cards currently link with `property.id`. Public URLs should use `property.slug` which is human-readable and already unique (Prisma `@unique` constraint). The private `(app)/listings/[slug]` route also uses slug — this is consistent.

- **Placing `(public)` layout inside `(app)`:** The `(public)` route group must be a sibling to `(app)`, not nested inside it. Both live directly under `app/`.

- **Putting the sidebar in the public layout:** The public layout must NOT include `SidebarProvider`, `AppSidebar`, or `HeaderBar`. It should only include `LandingNavbar` + `children` + `LandingFooter`.

- **Forgetting `force-light` class on public page:** The existing landing page uses `className="force-light min-h-screen bg-background"`. Check `globals.css` to see if `force-light` is a defined utility. If so, apply the same class to the public layout wrapper to prevent dark mode on the public page (matching the landing page look).

- **Not updating `AccessMarket` separately:** `FinalCTA` is automatically fixed when `LandingPropertyCard` is updated (it uses LandingPropertyCard). But `AccessMarket` has its own `PropertyRow` with a hardcoded `/listings/${property.id}` href — it must be updated independently.

- **Keeping the offer sidebar on the public page:** The OfferSidebar (with tier pricing) is NOT rendered on the public page. Tier pricing is one of the gated sections shown behind the `LockedSection` overlay.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Blur overlay for gated content | Custom CSS filter on individual elements | A single overlay `div` with `absolute inset-0 backdrop-blur-sm bg-background/60` | One overlay div is simpler and more reliable than blurring individual children |
| A separate "public" property data fetcher | New Prisma query or separate DB table | `getPropertyBySlug` from `lib/data/index.ts` | The same query returns all fields; just choose which fields to render vs. gate |
| Custom nav for public page | New navbar component from scratch | Reuse `LandingNavbar` + `LandingFooter` | Already built and styled for the public/marketing context |
| Animated section reveal on public page | Re-implement `SectionReveal` | Omit or reuse `SectionReveal` | Keep the public page simple — no need for reveal animations on a detail page |

**Key insight:** The entire public page is achieved through selective rendering and a UI overlay — no new data layer, no new auth system, no middleware. The "gating" is purely presentational.

## Common Pitfalls

### Pitfall 1: Route Group Layout Inheritance

**What goes wrong:** Developer assumes `(public)/layout.tsx` replaces the root `app/layout.tsx`. It does not — it nests inside it. Both run.

**Why it happens:** Confusion about how layout nesting works. The root `app/layout.tsx` always runs for every route. `(public)/layout.tsx` is an additional layer that wraps only routes in that group.

**How to avoid:** `(public)/layout.tsx` should NOT include `<html>` or `<body>` tags — those are in `app/layout.tsx`. It should just render a wrapper div + LandingNavbar + children + LandingFooter.

**Warning signs:** "You are using an unsupported combination of root layouts and layouts" error, or doubled `<html>` tags in the DOM.

### Pitfall 2: ID vs. Slug in Landing Page Links

**What goes wrong:** Landing page cards continue to link to `/listings/${property.id}` instead of `/p/${property.slug}`, sending users to the authenticated private page.

**Why it happens:** The landing page `page.tsx` maps properties to a subset that includes `id` but currently does NOT include `slug`. The child components use `property.id` for the href.

**How to avoid:** Add `slug: p.slug` to the property mapping in `app/page.tsx`, update the Property type in each affected component, and update all hrefs from `/listings/${id}` to `/p/${slug}`.

**Warning signs:** Clicking a landing page card navigates to the sidebar app view instead of the public listing view.

### Pitfall 3: LandingNavbar Scroll Behavior on Public Page

**What goes wrong:** `LandingNavbar` flips from dark to light mode based on `window.scrollY > window.innerHeight * 0.85` — this threshold assumes a fullscreen video hero. The public listing page has no fullscreen hero, so the navbar stays permanently dark.

**Why it happens:** The scroll threshold is calibrated for the landing page hero height, not an arbitrary detail page.

**How to avoid:** On the public listing page, the navbar should start in its light mode state immediately (or use a lower threshold). Options:
1. Pass a prop to `LandingNavbar` to control initial mode (requires editing the component)
2. Create a minimal `PublicNavbar` that is always in light mode (simpler, avoids modifying existing component)
3. The simplest option: just use `LandingNavbar` as-is — the user scrolling past the hero will trigger light mode anyway. If it feels wrong, adjust the threshold or add a prop.

**Recommendation:** Use `LandingNavbar` unchanged initially. If the dark-by-default state is jarring, add a `defaultLight?: boolean` prop to `LandingNavbar` to skip the scroll-based toggle.

**Warning signs:** Public page loads with a very dark/invisible navbar because the scroll threshold hasn't been reached.

### Pitfall 4: `force-light` Class Availability

**What goes wrong:** The landing page uses `className="force-light ..."` but this class may not be defined in `globals.css` or Tailwind config, causing it to be a no-op.

**Why it happens:** `force-light` is likely a custom utility defined in `globals.css` that forces the color scheme to light mode regardless of user's system preference or `next-themes` state.

**How to avoid:** Check `app/globals.css` before using `force-light`. If defined, apply it to the public layout wrapper. If not defined, omit it (or define it).

**Warning signs:** Public listing page renders in dark mode when user has dark mode set.

### Pitfall 5: `PropertyMeta` Field Shape

**What goes wrong:** `PropertyMeta` accepts `{ name, category, location, availability }`. The `location` field is from `property.location` and `availability` from `property.availability`. These fields exist in the Prisma schema — just verify the component receives them correctly.

**Why it happens:** The private detail page passes `property={property}` which includes all fields. The public page does the same — no mismatch expected, but worth confirming field names.

**How to avoid:** Pass `property={property}` directly — the Prisma `Property` model includes `name`, `category`, `location`, `availability` as confirmed in the schema.

## Open Questions

1. **LandingNavbar dark-by-default on public page**
   - What we know: `LandingNavbar` uses `window.scrollY > window.innerHeight * 0.85` to flip from dark to light. Public listing page has no fullscreen hero, so this threshold won't be crossed until after significant scrolling.
   - What's unclear: Whether the product design wants the navbar to start light on the public listing page.
