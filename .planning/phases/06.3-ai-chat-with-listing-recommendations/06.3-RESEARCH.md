**Researched:** 2026-02-22

**Confidence:** HIGH (all critical claims verified by codebase inspection; AI SDK verified via installed package types)

## Summary

Phase 6.3 adds the "Anvara AI" tab — a functional chat interface at `/ai` where users describe what they're looking for and get conversational AI responses along with up to 3 relevant listing recommendations rendered as `PropertyCard` components inside the chat.

The good news: almost everything needed already exists in the codebase. The AI infrastructure (`generateText` with `openai("o3-mini")`, the API route pattern), the chat UI patterns (message bubbles, typing indicator, scroll-to-bottom, input with send button), the sidebar nav entry (`{ title: "Anvara AI", href: "/ai", icon: Sparkles }` is already registered in `app-sidebar.tsx`), and the data layer (`getAllProperties()` returning all seed listings with tags, categories, descriptions) are all in place. The `PropertyCard` component at `app/(app)/listings/_components/property-card.tsx` accepts a `Property` type and renders fully with image, name, price, and link.

The only genuinely new work is: (1) `app/(app)/ai/page.tsx` — a Server Component shell, (2) `components/ai-chat/ai-chat-client.tsx` — a client component that manages conversation state purely in `useState` (no DB, no persistence), (3) `app/api/ai-chat/route.ts` — a new POST handler that receives the full conversation history and user message, calls `generateText` with a system prompt instructing the AI to recommend properties, and returns both the AI response text and an array of up to 3 recommended property slugs, (4) rendering `PropertyCard` components inline in the chat when a response includes recommendations.

**Primary recommendation:** Mirror the Phase 5 messaging pattern exactly — `useState` for chat messages on the client, `fetch` to a new `/api/ai-chat` route, `generateText` with `o3-mini` on the server. Pass all seed property data as context in the system prompt (or as structured JSON the AI can reference) so it can make grounded recommendations without any retrieval step.

### Pattern 1: Session-Only Conversation State

No database, no persistence. Chat history lives in React state and resets on page navigation.

```typescript
// components/ai-chat/ai-chat-client.tsx
"use client"

import { useState, useRef, useEffect } from "react"

type UserMessage = { role: "user"; content: string }
type AIMessage = { role: "assistant"; content: string; recommendedSlugs?: string[] }
type ChatMessage = UserMessage | AIMessage

export function AIChatClient() {
  const [messages, setMessages] = useState<ChatMessage[]>([])
  const [inputValue, setInputValue] = useState("")
  const [isLoading, setIsLoading] = useState(false)
  const scrollRef = useRef<HTMLDivElement>(null)

  useEffect(() => {
    if (scrollRef.current) {
      scrollRef.current.scrollTop = scrollRef.current.scrollHeight
    }
  }, [messages, isLoading])

  async function handleSend(content: string) {
    if (!content.trim() || isLoading) return

    const userMsg: UserMessage = { role: "user", content: content.trim() }
    const updatedMessages = [...messages, userMsg]

    setInputValue("")
    setMessages(updatedMessages)
    setIsLoading(true)

    try {
      const response = await fetch("/api/ai-chat", {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({ messages: updatedMessages }),
      })
      const data = await response.json()

      if (response.ok) {
        const aiMsg: AIMessage = {
          role: "assistant",
          content: data.text,
          recommendedSlugs: data.recommendedSlugs ?? [],
        }
        setMessages((prev) => [...prev, aiMsg])
      }
    } catch {
      // Handle error
    } finally {
      setIsLoading(false)
    }
  }
}
```

**Why:** Exact same pattern as Phase 5's `ConversationView.handleSend`. No new patterns introduced.

### Pattern 2: API Route with Property Catalog in System Prompt

The route receives the full conversation history and sends it to o3-mini along with a system prompt that includes all property listings as structured context.

```typescript
// app/api/ai-chat/route.ts
import { generateText } from "ai"
import { openai } from "@ai-sdk/openai"
import { getAllProperties } from "@/lib/data"

export const maxDuration = 60

type ChatMessage = { role: "user" | "assistant"; content: string }

function buildSystemPrompt(properties: Awaited<ReturnType<typeof getAllProperties>>): string {
  const catalog = properties
    .map((p) => ({
      slug: p.slug,
      name: p.name,
      category: p.category,
      subcategory: p.subcategory,
      location: p.location,
      tags: p.tags.join(", "),
      priceFrom: p.priceFrom,
      audienceAgeRange: p.audienceAgeRange,
      description: p.description.slice(0, 200),
    }))

  return `You are Anvara AI, a sponsorship marketplace assistant. Help advertisers find relevant sponsorship opportunities.

Available listings (JSON):
${JSON.stringify(catalog, null, 2)}

When responding:
1. Engage conversationally with the user's stated goals
2. Recommend UP TO 3 relevant listings by including their slugs in your response
3. Return your response as JSON: { "text": "your conversational response", "recommendedSlugs": ["slug1", "slug2"] }
4. If no listings are relevant, return an empty recommendedSlugs array
5. Keep responses concise and actionable (2-3 sentences)
6. Reference specific property names and categories in your response`
}

export async function POST(request: Request) {
  try {
    const body = await request.json()
    const { messages } = body as { messages: ChatMessage[] }

    const properties = await getAllProperties()
    const systemPrompt = buildSystemPrompt(properties)

    const { text } = await generateText({
      model: openai("o3-mini"),
      system: systemPrompt,
      messages,
    })

    // Parse AI response — AI is instructed to return JSON
    let parsed: { text: string; recommendedSlugs: string[] }
    try {
      parsed = JSON.parse(text)
    } catch {
      // Fallback if AI doesn't return valid JSON
      parsed = { text, recommendedSlugs: [] }
    }

    // Validate that slugs exist in our catalog
    const validSlugs = new Set(properties.map((p) => p.slug))
    const recommendedSlugs = (parsed.recommendedSlugs ?? [])
      .filter((s) => validSlugs.has(s))
      .slice(0, 3)

    return Response.json({ text: parsed.text, recommendedSlugs })
  } catch (err) {
    console.error("[POST /api/ai-chat]", err)
    return Response.json({ error: "Failed to generate response" }, { status: 500 })
  }
}
```

**Why:** Structured JSON response from the AI is the simplest way to get both text and slugs without any natural-language parsing. o3-mini follows JSON format instructions reliably.

### Pattern 3: Rendering PropertyCard in Chat Messages

The AI message component renders the text response and conditionally renders PropertyCard components when slugs are present.

```typescript
// components/ai-chat/ai-message.tsx
import { PropertyCard } from "@/app/(app)/listings/_components/property-card"
import type { getAllProperties } from "@/lib/data"

type Property = Awaited<ReturnType<typeof getAllProperties>>[number]

interface AIMessageProps {
  content: string
  recommendedProperties: Property[]
}

export function AIMessage({ content, recommendedProperties }: AIMessageProps) {
  return (
    <div className="flex justify-start">
      <div className="flex flex-col gap-3 max-w-[85%]">
        {/* Text bubble */}
        <div className="bg-muted text-foreground rounded-2xl rounded-bl-sm px-4 py-2.5">
          <p className="text-sm leading-relaxed">{content}</p>
        </div>
        {/* Recommendation cards — grid */}
        {recommendedProperties.length > 0 && (
          <div className="grid grid-cols-1 gap-3 sm:grid-cols-3">
            {recommendedProperties.map((property) => (
              <PropertyCard key={property.id} property={property} />
            ))}
          </div>
        )}
      </div>
    </div>
  )
}
```

**Why:** PropertyCard already accepts the full property object from `getAllProperties()`. The client must hold a lookup map of slug → property to resolve slugs returned by the API. The page passes the full property list as a prop to `AIChatClient`, which builds the map.

### Pattern 4: Property Resolution on the Client

The client component receives all properties as a prop (fetched server-side in page.tsx) and resolves slugs to property objects when rendering AI messages.

```typescript
// app/(app)/ai/page.tsx
import { getAllProperties } from "@/lib/data"
import { AIChatClient } from "@/components/ai-chat/ai-chat-client"

export const metadata = { title: "Anvara AI | Anvara" }

export default async function AIPage() {
  const properties = await getAllProperties()
  return <AIChatClient properties={properties} />
}
```

```typescript
// In AIChatClient — resolve slugs to property objects for rendering
const propertyMap = useMemo(
  () => new Map(properties.map((p) => [p.slug, p])),
  [properties]
)

// When rendering an AI message with recommendedSlugs:
const recommendedProperties = aiMsg.recommendedSlugs
  .map((slug) => propertyMap.get(slug))
  .filter(Boolean)
```

**Why:** RSC page fetches properties once on initial render. The client has the full catalog for slug resolution without additional API calls. The API route also fetches properties independently (it can't receive the client's prop). This is an acceptable double-fetch since `getAllProperties()` is a fast Prisma query against seeded data.

### Anti-Patterns to Avoid

- **Calling `getAllProperties()` in a client component:** All DB access must go through server components or API routes. Properties for rendering are fetched in `page.tsx` (RSC) and passed as props.
- **Persisting conversations to the database:** The spec says "within the session" only. Adding DB writes would require schema changes and is explicitly out of scope.
- **Trying to use `useChat` from `@ai-sdk/react`:** This package is not installed. The existing manual `useState` + `fetch` pattern is proven, installed, and sufficient.
- **Copying PropertyCard:** Import from its existing path. Do not duplicate it.
- **Rendering raw AI JSON in the text bubble:** Always parse the JSON and display only `parsed.text` to the user — never show the raw `{ "text": ..., "recommendedSlugs": ... }` string.
- **Trusting AI-returned slugs without validation:** Always filter recommended slugs against the known property catalog before rendering.

## Common Pitfalls

### Pitfall 1: AI Returns Non-JSON Despite Instruction

**What goes wrong:** o3-mini occasionally returns prose instead of JSON when the system prompt instruction is unclear or the user message confuses it.
**Why it happens:** o3-mini is a reasoning model that sometimes "thinks out loud" before emitting the JSON structure.
**How to avoid:** Wrap JSON.parse in try/catch and fall back to `{ text: rawResponse, recommendedSlugs: [] }` so the chat never breaks. The user still gets a response — just without recommendations.
**Warning signs:** TypeError in the console when parsing; empty recommendation slots.

### Pitfall 2: PropertyCard Requires `Property` Type with `packages` Included

**What goes wrong:** PropertyCard accesses `property.tags` and other fields from the full `getAllProperties()` return type. If slugs are resolved from a subset of property fields (e.g., a lighter query), TypeScript will error.
**Why it happens:** PropertyCard uses `type Property = Awaited<ReturnType<typeof getAllProperties>>[number]` — it expects the full shape including `packages[]`.
**How to avoid:** Always pass properties fetched via `getAllProperties()` (which includes `packages: true`). Do not create a lighter query for this phase.
**Warning signs:** TypeScript error on `property.packages` or `property.tags` in PropertyCard.

### Pitfall 3: Double Fetch of getAllProperties

**What goes wrong:** Properties are fetched twice — once in `page.tsx` (RSC) and once in `/api/ai-chat/route.ts` for every AI call.
**Why it happens:** The RSC needs properties to pass as props for client-side slug resolution; the API route needs them to build the system prompt.
**Why it's acceptable:** The seed data is deterministic and Prisma returns it very fast. This is a prototype, not production. Both fetches hit the same Prisma instance.
**Alternative if needed:** Cache `getAllProperties()` with `unstable_cache` from `next/cache` — but this is premature optimization for this phase.

### Pitfall 4: maxDuration Not Set on New Route

**What goes wrong:** The `/api/ai-chat` route times out after 10 seconds (Vercel default) because o3-mini can take 10-30 seconds for reasoning.
**Why it happens:** Same issue as Phase 5's `/api/chat` route.
**How to avoid:** Export `export const maxDuration = 60` at the top of `/api/ai-chat/route.ts`. This is already done in `/api/chat/route.ts` — copy the pattern.
**Warning signs:** Route returns 504 after 10 seconds.

### Pitfall 5: Sidebar Nav Already Has "Anvara AI" — Don't Add It

**What goes wrong:** Adding a duplicate nav entry or trying to add the icon.
**Why it happens:** Researcher might assume the nav item doesn't exist yet.
**Reality:** The sidebar already has:
```typescript
{ title: "Anvara AI", href: "/ai", icon: Sparkles },
```
in `components/app-sidebar.tsx`. The nav item IS there. Creating `app/(app)/ai/page.tsx` is all that's needed to make it work — no sidebar changes required.
**Warning signs:** Planning tasks that include "update sidebar nav" — this is already done.

### Pitfall 6: Chat History Accumulation in System Prompt

**What goes wrong:** In long conversations, passing the full message history alongside the full property catalog in the system prompt can hit o3-mini's context limits.
**Why it happens:** Each turn sends all prior messages + the full catalog JSON.
**How to avoid:** For a demo prototype, this is unlikely to be an issue (sessions are short, catalog is ~10-15 properties). If needed, use `pruneMessages` from the `ai` SDK to trim older messages. Not needed for this phase.

## Open Questions

1. **JSON Reliability from o3-mini**
   - What we know: o3-mini follows JSON instructions well in practice; the Phase 5 system prompt shows the model handles structured instructions reliably
   - What's unclear: Whether the reasoning model "leaks" reasoning tokens before the JSON in certain edge cases
   - Recommendation: Implement the try/catch fallback (show text without recommendations on parse failure) — this handles all edge cases gracefully

2. **PropertyCard Image Rendering in Chat**
   - What we know: PropertyCard uses `next/image` with `sizes="(max-width: 768px) 50vw, (max-width: 1024px) 33vw, 20vw"` — sized for a browse grid
   - What's unclear: Whether the sizes hint is appropriate when rendered in a narrower chat message column
   - Recommendation: PropertyCard can be used as-is; image is `aspect-[2/1]` and will fill whatever column width it's in. The `sm:grid-cols-3` layout in the AI message component means each card is ~1/3 of the chat width on desktop — acceptable.

3. **Conversation Context Passing**
   - What we know: The client sends `messages` (user + assistant turns) to the API route on each send
   - What's unclear: Whether assistant messages should include the full JSON (with recommendedSlugs) or only the text content in the history
   - Recommendation: Store assistant messages in client state with only the `content` text (not the JSON blob). When sending history to the API route, convert AI messages to `{ role: "assistant", content: msg.content }` — the prior recommendations don't need to be re-sent as context.
