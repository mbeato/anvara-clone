# Phase 6.4: Stub Pages and Platform Tour - Research

**Researched:** 2026-02-22
**Domain:** Next.js 15 page stubs, ShadCN Dialog multi-step tour, sidebar modal integration
**Confidence:** HIGH

## Summary

This phase has two distinct workstreams: (1) creating three stub pages — Campaigns, Deals, and Personalize — with mock data that feels realistic, and (2) building a multi-step "Show Me Around" modal tour triggered by the sidebar footer item instead of navigating to a `/tour` route.

The codebase already has all the patterns and components needed. The dashboard provides a complete reference: Server Component page → deterministic data generation in `lib/data/analytics.ts` → client-side chart/table components. Campaigns and Deals can follow this pattern exactly with new deterministic mock data sets. Personalize can be a settings-style form using existing ShadCN primitives.

The tour requires a small but important architectural decision: the `/tour` sidebar item currently uses `Link href="/tour"` with `asChild`, which would 404 since no page exists. It must become a button that opens a Dialog. Since `AppSidebar` is already a `"use client"` component (it uses `usePathname`), adding `useState` for the modal open state is straightforward — no new client boundary required. The multi-step logic is plain React state (`step` index), not a library.

**Primary recommendation:** Implement all four items without adding new npm packages. Use existing ShadCN Dialog for the tour modal, existing Card/Badge/Table patterns for stub pages, and deterministic inline mock data (same pattern as `analytics.ts`).

### Pattern 2: Tour Modal — State in AppSidebar (No New Client Boundary)

**What:** The `Show Me Around` footer item changes from `Link asChild` to a `SidebarMenuButton` with `onClick` that sets a state variable. A `TourModal` component is rendered adjacent to the sidebar menu with its `open` prop controlled by that state.

**Why this works:** `AppSidebar` is already `"use client"` (uses `usePathname`). Adding `useState` and rendering a Dialog inside it requires no boundary changes.

**Key implementation note:** `SidebarMenuButton` renders as a `<button>` by default when `asChild` is NOT provided. The current `/tour` item uses `asChild` with a `Link` child. Remove `asChild` and `Link`, pass `onClick` instead. Keep tooltip and isActive props.

**Example:**
```typescript
// Source: components/app-sidebar.tsx pattern + components/ui/dialog.tsx pattern
```

### Pattern 3: Multi-Step Tour Modal — React State Only

**What:** A client component that maintains a `step` index (0-based). Navigation buttons increment/decrement. `DialogContent` renders step-specific content based on index.

**When to use:** For walkthroughs with 3-6 discrete slides. Simpler than a library.

**Example (TourModal):**
```typescript
// Source: components/ui/dialog.tsx + React useState pattern
"use client"

import { useState } from "react"
import { Dialog, DialogContent, DialogHeader, DialogTitle, DialogDescription, DialogFooter } from "@/components/ui/dialog"
import { Button } from "@/components/ui/button"

const STEPS = [
  {
    title: "Welcome to Anvara",
    description: "Your sponsorship marketplace for connecting advertisers with event properties.",
    // icon or illustration...
  },
  {
    title: "Browse the Marketplace",
    description: "Discover thousands of sponsorship opportunities across festivals, sports, food events and more.",
  },
  {
    title: "Launch Campaigns",
    description: "Track your sponsorships, manage spend, and measure reach in one dashboard.",
  },
  {
    title: "AI-Powered Discovery",
    description: "Use Anvara AI to find perfect-fit properties using natural language search.",
  },
]

interface TourModalProps {
  open: boolean
  onOpenChange: (open: boolean) => void
}

export function TourModal({ open, onOpenChange }: TourModalProps) {
  const [step, setStep] = useState(0)

  const currentStep = STEPS[step]
  const isFirst = step === 0
  const isLast = step === STEPS.length - 1

  function handleClose() {
    onOpenChange(false)
    // Reset step when closed so next open starts fresh
    setStep(0)
  }

  return (
    <Dialog open={open} onOpenChange={handleClose}>
      <DialogContent className="sm:max-w-md">
        <DialogHeader>
          <DialogTitle>{currentStep.title}</DialogTitle>
          <DialogDescription>{currentStep.description}</DialogDescription>
        </DialogHeader>

        {/* Step content area */}
        {/* ... */}

        {/* Progress dots */}
        <div className="flex justify-center gap-1.5 py-2">
          {STEPS.map((_, i) => (
            <span
              key={i}
              className={`size-2 rounded-full ${i === step ? "bg-primary" : "bg-muted"}`}
            />
          ))}
        </div>

        <DialogFooter className="gap-2 sm:gap-0">
          {!isFirst && (
            <Button variant="outline" onClick={() => setStep(s => s - 1)}>Back</Button>
          )}
          {isLast ? (
            <Button onClick={handleClose}>Get Started</Button>
          ) : (
            <Button onClick={() => setStep(s => s + 1)}>Next</Button>
          )}
        </DialogFooter>
      </DialogContent>
    </Dialog>
  )
}
```

### Anti-Patterns to Avoid

- **Do not create a `/tour` page.tsx**: The sidebar item must trigger the modal, not navigate. A `/tour` page will 404 and is the wrong pattern — the tour is an overlay, not a route.
- **Do not use Math.random() in stub data**: Existing codebase enforces deterministic data generation. Use hardcoded arrays or prime-based arithmetic.
- **Do not add react-joyride or shepherd.js**: These are element-spotlight libraries for annotating real UI. The requirement is a modal walkthrough, which is simply a stepped Dialog. Library adds ~50KB for no benefit.
- **Do not pass iconName strings for the tour steps**: Tour content stays within a single client component. `LucideIcon` can be referenced directly in the STEPS array (it doesn't cross a server/client boundary here).
- **Do not make Personalize async**: No DB queries needed. A plain `export default function` (not `async`) is simpler.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Step progress indicator | Custom CSS animation | Simple `bg-primary`/`bg-muted` dot pattern | Tour has ≤6 steps; dots are idiomatic and minimal |
| Modal accessibility | Manual focus trap, aria | Radix Dialog (`DialogContent`) | Already handles focus trap, aria-labelledby, scroll lock |
| Sidebar button vs link | Custom routing hack | Remove `asChild`/`Link`, use `onClick` | SidebarMenuButton renders as `<button>` by default without asChild |
| Campaign status colors | Custom badge class | `Badge variant="default"/"outline"/"secondary"` | Existing Badge has all needed variants |

**Key insight:** The existing ShadCN components cover 100% of this phase's UI needs. Zero new components need to be built from scratch.

## Common Pitfalls

### Pitfall 1: Tour Step Reset on Dialog Close
**What goes wrong:** If `step` state is not reset when the modal closes, reopening the tour starts at the last viewed step, not step 0.
**Why it happens:** Dialog's `onOpenChange` is called with `false` on close. If only `onOpenChange(false)` is called without also calling `setStep(0)`, the state persists.
**How to avoid:** In the `handleClose` function, call both `onOpenChange(false)` and `setStep(0)`. Or use `useEffect` to reset on `open` becoming `false`.
**Warning signs:** "Next" button on last step stays pressed after reopen.

### Pitfall 2: SidebarMenuButton Not Rendering as Button
**What goes wrong:** Keeping `asChild` on the "Show Me Around" item while passing `onClick` — Radix Slot forwards props to the child element, but if the child is a `Link`, clicking navigates instead of opening the modal.
**Why it happens:** `asChild` + `Link` = the Link handles the click (navigation wins).
**How to avoid:** Remove `asChild` and the `Link` child from the "Show Me Around" item. Pass `onClick` directly to `SidebarMenuButton` with no `asChild`. It renders as a plain `<button>`.
**Warning signs:** Clicking "Show Me Around" navigates to `/tour` (404).

### Pitfall 3: Wrapping TourModal Outside Sidebar JSX
**What goes wrong:** Placing `<TourModal>` inside `<Sidebar>` element causes z-index or portal layering issues.
**Why it happens:** Sidebar uses sticky positioning; Dialog portal renders to `document.body`. If the modal is nested inside a positioned ancestor, the fixed overlay may clip.
**How to avoid:** Render `<TourModal>` as a sibling to `<Sidebar>`, wrapped in a `<>` fragment at the top level of `AppSidebar`'s return.
**Warning signs:** Modal appears behind sidebar or overlay doesn't cover full screen.

### Pitfall 4: Deterministic Data That Looks Too Uniform
**What goes wrong:** Hardcoded stub data with identical-looking numbers reads as fake immediately (e.g., every campaign has exactly $10,000 spend).
**Why it happens:** Fast copy-paste when writing mock arrays.
**How to avoid:** Vary values realistically. Campaigns: mix of active/paused/completed status, different dates, varying spend ($3K–$45K range), varying impressions. Use distinctive event names from the seeded properties already in the DB (Coachella, Austin Food & Wine, etc.) to feel connected to existing data.
**Warning signs:** Numbers are round or sequential (1000, 2000, 3000).

### Pitfall 5: Missing `loading.tsx` for Stub Pages
**What goes wrong:** Stub pages load without Suspense fallback, showing blank content during hydration.
**Why it happens:** Dashboard has `loading.tsx` but it's easy to forget for new pages.
**How to avoid:** Add a simple `loading.tsx` per page using `Skeleton` components matching the page layout. Dashboard's `loading.tsx` is the reference pattern.
**Warning signs:** Flash of empty content on navigation.

## Open Questions

1. **How many tour steps?**
   - What we know: Phase context says "a couple steps" — suggests 2-5
   - What's unclear: Exact content/screens to showcase
   - Recommendation: Use 4 steps covering Welcome, Marketplace, Dashboard, AI. Feels complete without being tedious. Each step has icon + title + description + optional CTA link.

2. **Should Personalize have any functional controls?**
   - What we know: No user preferences model exists in Prisma schema. Phase says "placeholder content."
   - What's unclear: Whether sliders/toggles should be interactive (client) or purely visual (static HTML)
   - Recommendation: Make form controls rendered as disabled/static — they look real but submit nothing. Avoids needing a client boundary on the Personalize page.

3. **Should Deals show a Kanban board or a table?**
   - What we know: The phase context says "mock data tables/cards." Deal pipelines are commonly shown as Kanban (columns by stage) or as a table.
   - What's unclear: User preference
   - Recommendation: Use a stage-grouped card layout (3-4 columns: Prospecting / Negotiating / Agreed / Live) rather than a flat table. More visually distinctive from Campaigns which already uses a table. Columns can be built with CSS grid, no Kanban library needed.

4. **Does `/tour` need a redirect?**
   - What we know: The sidebar "Show Me Around" will no longer link to `/tour`. No `app/(app)/tour/page.tsx` exists or will be created.
   - What's unclear: Whether anything else links to `/tour`
   - Recommendation: Don't create a redirect. The sidebar item change removes the only entry point to `/tour`. A 404 at `/tour` is acceptable since users can't navigate there through the UI.
